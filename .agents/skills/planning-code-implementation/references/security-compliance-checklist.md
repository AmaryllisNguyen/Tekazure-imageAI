# Security Compliance Checklist (Mandatory)

Use this checklist as a required pass/fail gate for any output generated by `planning-code-implementation`.

## Rules and Evidence

| Rule | Pass Criteria | Evidence Expected |
|---|---|---|
| No hardcoded secrets / no key exposure | No credentials in code snippets, logs, responses, or frontend payloads | Secret scan result and explicit statement in plan |
| Secrets from `.env` or secret manager only | Config loading path is documented and environment-specific | Config section naming source (`.env`/manager) and import flow |
| RLS enabled when DB supports it | RLS enablement and policy verification are defined in rollout | DDL/config step + test cases proving enforcement |
| Backend re-validation of all input | Validation exists at server boundary independent of frontend checks | Schema + semantic validation steps in API plan |
| Dependency hygiene enforced | Dependency audit cadence and update policy are defined | Tooling, cadence (e.g., weekly), and CVE response SLA |
| Security middleware covers protected routes | Route groups have mapped middleware with no uncovered protected path | Route-to-middleware coverage map and gap resolution |

## Quick QA Gate

Before approving:
- All six rules must be `Pass`.
- Any `Fail` blocks release recommendation.
- If rule is not applicable, provide explicit justification and approving authority.

## Common Failure Patterns

- Examples include real API keys or env values.
- Validation described only at frontend layer.
- Protected routes missing auth/rate-limit middleware.
- Dependencies pinned but never audited.
- Database used in production without RLS verification.
